# Git 操作规范 —— NUDT 自瞄培训

> 所有作业通过 **Fork + Pull Request** 提交。请严格按本文档操作，遇到问题先翻文末「常见问题」。

## 〇、准备工作（只需做一次）

### 1. 注册 GitHub 账号并配置 SSH 密钥

作业仓库用 SSH 方式（`git@github.com:...`）拉取，需要先配好密钥，否则 clone/push 会失败。

```bash
# 生成密钥（邮箱填你的 GitHub 注册邮箱）
ssh-keygen -t ed25519 -C "你的邮箱"
# 一路回车即可

# 查看公钥内容，复制输出
cat ~/.ssh/id_ed25519.pub
```

把复制的内容粘贴到 GitHub：右上角头像 → Settings → SSH and GPG keys → New SSH key。

```bash
# 测试是否配好，看到 successfully authenticated 即成功
ssh -T git@github.com
```

### 2. 配置你的身份信息

```bash
git config --global user.name "你的姓名"
git config --global user.email "你的GitHub邮箱"
```

## 一、Fork 主仓库（只需做一次）

打开主仓库页面：

```
https://github.com/Winter-Raymond/nudt-rm-auto-aim-training
```

点右上角 **Fork** 按钮，把仓库复制到你自己的账号下。完成后你会拥有一个属于自己的副本：

```
github.com/你的用户名/nudt-rm-auto-aim-training
```

> Fork 出来的这份是你的私人空间，怎么改都不会影响主仓库。作业全在自己的 fork 里做。

## 二、Clone 到本地（只需做一次）

把**你自己的 fork**（注意是你的用户名，不是 Winter-Raymond）克隆下来：

```bash
git clone git@github.com:你的用户名/nudt-rm-auto-aim-training.git
cd nudt-rm-auto-aim-training
```

### 配置上游仓库（upstream）

为了以后能同步主仓库的更新（比如老师发布了新作业），添加主仓库为 `upstream`：

```bash
git remote add upstream git@github.com:Winter-Raymond/nudt-rm-auto-aim-training.git

# 检查一下，应该能看到 origin（你的fork）和 upstream（主仓库）
git remote -v
```

正确的结果长这样：

```
origin    git@github.com:你的用户名/nudt-rm-auto-aim-training.git (fetch)
origin    git@github.com:你的用户名/nudt-rm-auto-aim-training.git (push)
upstream  git@github.com:Winter-Raymond/nudt-rm-auto-aim-training.git (fetch)
upstream  git@github.com:Winter-Raymond/nudt-rm-auto-aim-training.git (push)
```

## 三、做每一次作业的标准流程

### 第 1 步：同步主仓库最新内容

每次开始新作业前，先把主仓库的更新拉到本地，避免落后。

```bash
git checkout main
git fetch upstream
git merge upstream/main
git push origin main      # 顺便把你的 fork 也更新一下
```

### 第 2 步：新建一个作业分支

**不要直接在 main 上改代码**。每次作业开一个独立分支，分支名见命名规范。

```bash
git checkout -b week1-git-basics
```

分支命名规范：`周次-作业主题`，全小写，用连字符分隔。例如：

- `week1-git-basics`
- `week2-opencv`
- `week3-armor-detection`

### 第 3 步：写代码

在对应的作业目录里完成你的代码，比如 `homework/week1-git-and-cpp/你的用户名/`。

> 每个人在自己的用户名子目录下提交，避免互相覆盖。

### 第 4 步：提交（commit）

```bash
# 查看改了哪些文件
git status

# 添加你要提交的文件（建议指定目录，别用 git add . 全加）
git add homework/week1-git-and-cpp/你的用户名/

# 提交，写清楚做了什么
git commit -m "feat(week1): 完成 Git 流程练习与 C++ 基础题"
```

### Commit message 规范

格式：`类型(范围): 简短描述`

| 类型 | 含义 |
|---|---|
| `feat` | 新增功能 / 完成作业 |
| `fix` | 修复 bug |
| `docs` | 文档改动 |
| `refactor` | 重构，不改功能 |

示例：

```
feat(week3): 实现装甲板灯条检测
fix(week3): 修正灯条角度筛选的阈值错误
```

### 第 5 步：推送到你的 fork

```bash
git push origin week1-git-basics
```

### 第 6 步：发起 Pull Request

1. 打开你的 fork 页面，会出现一个 **Compare & pull request** 的提示按钮，点它。
2. 确认合并方向是：

   ```
   base: Winter-Raymond/nudt-rm-auto-aim-training  main
        ←
   head: 你的用户名/nudt-rm-auto-aim-training  week1-git-basics
   ```

   也就是把**你的分支**合并进**主仓库的 main**。

3. PR 标题写清楚，比如：`[week1] 张三 - Git流程与C++基础`
4. 按模板填写说明（做了什么、遇到什么问题），提交。
5. 提交后等老师 review。如果老师提了修改意见，**不用重新发 PR**，直接在本地同一个分支改、提交、push，PR 会自动更新：

   ```bash
   # 在 week1-git-basics 分支上修改后
   git add .
   git commit -m "fix(week1): 根据review修改XXX"
   git push origin week1-git-basics
   ```

### 第 7 步：作业被合并后，清理分支

老师 merge 你的 PR 后，这个分支就完成使命了，可以删掉：

```bash
git checkout main
git branch -d week1-git-basics              # 删本地分支
git push origin --delete week1-git-basics   # 删 fork 上的分支
```

然后回到「第三步第 1 步」同步最新代码，开始下一次作业。

## 四、一次作业流程速查（背下来）

```bash
git checkout main && git fetch upstream && git merge upstream/main   # 同步
git checkout -b weekN-topic        # 开分支
# ... 写代码 ...

git commit -m "feat(weekN): xxx"   # 提交
git push origin weekN-topic        # 推送
# 网页发起 PR
```

## 五、常见问题（FAQ）

**Q1：`git push` 报 Permission denied（publickey）**
SSH 密钥没配好。回到「准备工作」重新配置，并用 `ssh -T git@github.com` 测试。

**Q2：我不小心在 main 分支上写了代码怎么办？**
还没 commit 的话，把改动转移到新分支：

```bash
git checkout -b week1-git-basics   # 带着未提交的改动切到新分支
```

已经 commit 在 main 上了，告诉老师或翻 git 文档用 `git reset`，别自己乱试。

**Q3：发起 PR 时显示一堆别人的提交 / 冲突？**
说明你的分支落后于主仓库太多。先同步再说：

```bash
git checkout main
git fetch upstream && git merge upstream/main
git checkout 你的分支
git merge main          # 把最新的 main 合进你的分支，解决冲突后再 push
```

**Q4：clone 很慢或失败？**
GitHub 网络问题。确认代理已开启。

**Q5：commit 后发现写错了 commit message？**
只要还没 push，可以改：

```bash
git commit --amend -m "新的正确信息"
```

## 六、铁律（违反了会很麻烦）

- **永远不在 `main` 上直接写代码**，每次作业都开新分支。
- **永远 clone 自己的 fork，不要 clone 主仓库**（主仓库你没写权限）。
- **只提交自己目录下的文件**，`git add` 时指定路径，别用 `git add .` 把编译产物、别人代码一起加进去。
- **编译产物不入库**（`build/`、`*.o`、`*.exe`），这些已经在 `.gitignore` 里忽略了。
- 提交前先 `git status` 看一眼改了啥。
